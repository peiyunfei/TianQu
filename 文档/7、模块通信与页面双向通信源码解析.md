# 模块通信与页面双向通信源码深度解析

在跨模块的大型组件化项目中，各业务模块通常处于解耦状态（即不相互依赖代码）。为了让模块之间互相调用（**模块通信**）以及页面之间传递返回值（**页面双向通信**），天衢 路由框架设计了两套优雅的机制。

本文将逐行剖析这两套机制的源码实现，深度讲解其背后的设计思想。

---

## 一、 模块通信源码解析 (ServiceLocator 模式)

在组件化架构中，A 模块不能直接 `new BServiceImpl()`，因为没有类的依赖。因此我们需要一个“服务注册与发现”中心，即 `ServiceManager`。天衢 采用了 **ServiceLocator（服务定位器）模式**。

### 1. 外观类 `ServiceManager.kt`

`ServiceManager` 是对外暴露的 Facade（门面模式），它使用 `object` 单例，隐藏了内部复杂的解析逻辑。

```kotlin
object ServiceManager {
    // 底层真实解析器，默认实现是 DefaultServiceResolver
    // 采用接口隔离，方便测试时注入 MockResolver 进行单元测试
    private var resolver: ServiceResolver = DefaultServiceResolver()

    /**
     * 初始化：将编译期 (KSP) 收集到的所有服务接口与实现的映射表注册进来
     * registryProviders: Map<KClass<*>, () -> Any> 
     * key 是接口的 Class，value 是一个无参的 Lambda (即对象创建工厂)
     */
    fun init(registryProviders: Map<KClass<*>, () -> Any>) {
        resolver.init(registryProviders)
    }

    /**
     * 同步获取服务实例，使用 inline + reified 语法糖，
     * 开发者调用时不需要传 T::class，直接写 ServiceManager.getService<UserService>()
     */
    inline fun <reified T : Any> getService(): T? {
        return getService(T::class) as? T
    }

    // ... 省略挂起函数 awaitService ...
}
```

### 2. 核心解析器 `DefaultServiceResolver.kt`

这是模块通信的核心脏，负责**懒加载实例化**、**单例缓存**和**并发安全**。

```kotlin
class DefaultServiceResolver : ServiceResolver {

    // 【1】存储接口与其实现类的"工厂方法"映射，来源于 KSP 自动生成
    private val providers = mutableMapOf<KClass<*>, () -> Any>()
    
    // 【2】同步实例缓存：如果服务已经创建过，直接返回这个实例，实现单例(Singleton)机制
    private val syncInstances = mutableMapOf<KClass<*>, Any>()
    
    // 【3】异步实例缓存：用于协程环境中，防止并发获取时创建多个实例
    private val deferredInstances = mutableMapOf<KClass<*>, Deferred<Any?>>()
    
    // 【4】互斥锁，保证多线程/多协程并发请求同一个服务时，严格保证只初始化一次
    private val mutex = Mutex()

    // 专属的协程作用域，运行在默认后台线程池 (Dispatchers.Default)，SupervisorJob 确保一个子协程失败不影响其他协程
    private val scope = CoroutineScope(SupervisorJob() + Dispatchers.Default)

    /**
     * 【同步获取服务】
     */
    override fun getService(clazz: KClass<*>): Any? {
        // 第一步：检查缓存。如果实例化过，直接返回。极大地提升了获取性能。
        if (syncInstances.containsKey(clazz)) {
            return syncInstances[clazz]
        }
        
        // 第二步：从 KSP 注册表中获取工厂 Lambda
        val provider = providers[clazz]
        if (provider != null) {
            // 第三步：执行 Lambda，即 provider()。
            // 💡 核心设计思想（懒加载与无反射）：
            // KSP 生成的代码是形如：{ UserServiceImpl() }
            // 所以这里直接执行普通方法创建了对象，没有任何反射带来的性能损耗！
            // 只有在第一次被 getService 请求时才会真正 new 这个对象（懒加载，优化启动速度和内存）。
            val instance = provider()
            
            // 第四步：放入同步缓存
            syncInstances[clazz] = instance
            // 第五步：同步状态到异步缓存，防止此时正好有异步请求在等待
            deferredInstances[clazz] = CompletableDeferred(instance)
            
            return instance
        }
        // 未找到对应的服务注册
        return null
    }

    /**
     * 【异步获取服务 (协程与并发安全)】
     */
    override fun <T : Any> getServiceAsync(clazz: KClass<T>): AsyncResult<T?> {
        val result = DeferredAsyncResult<T?>()
        
        scope.launch {
            // 使用 Mutex 互斥锁包围核心的"获取或创建"逻辑，保证绝对的线程安全
            val deferred = mutex.withLock {
                // 如果 deferredInstances 里没有该类，就执行大括号里的逻辑创建一个新的 Deferred 放入
                deferredInstances.getOrPut(clazz) {
                    // 二次检查同步缓存 (Double Check Locking 思想)
                    if (syncInstances.containsKey(clazz)) {
                        CompletableDeferred(syncInstances[clazz])
                    } else {
                        val provider = providers[clazz]
                        if (provider != null) {
                            // 启动一个新的后台异步任务(async)去实例化对象
                            scope.async {
                                val instance = provider()
                                syncInstances[clazz] = instance
                                instance // 返回结果
                            }
                        } else {
                            // 如果找不到服务，返回一个直接完结的 Deferred(null)
                            CompletableDeferred<Any?>().apply { complete(null) }
                        }
                    }
                }
            }
            
            try {
                // 等待对象创建完成（如果已经在创建中，这里会挂起等待；如果已经创建完，立刻继续）
                val resolvedInstance = deferred.await() as? T
                // 将真实实例通过 complete 推给调用方
                result.complete(resolvedInstance)
            } catch (e: Exception) {
                result.complete(null)
            }
        }
        return result
    }
}
```

### 3. 为什么 getServiceAsync 要开启新协程 (scope.launch / scope.async) 创建对象？

如果您仔细观察 `getServiceAsync` 的代码，会发现里面嵌套了 `scope.launch` 和 `scope.async`。为什么不直接创建对象？这是基于以下三个绝妙的架构考量：

1. **防止阻塞调用方线程（特别是主线程）**
   `getServiceAsync` 是一个**普通函数**（不是 suspend 函数），通常在 UI 线程中被调用。由于服务对象的初始化（即 `provider()` 的执行）可能非常耗时（例如：初始化数据库、读取本地文件、预加载重型资源），如果直接执行 `provider()` 就会卡死主线程导致掉帧或 ANR。因此，必须将其放入后台协程（`Dispatchers.Default`）中执行。
2. **桥接非挂起世界与挂起世界**
   为了实现并发安全，底层使用了 `mutex.withLock`（挂起函数）和 `deferred.await()`（挂起函数）。因为 `getServiceAsync` 本身不能挂起，所以它**必须**通过外层的 `scope.launch` 开启一个协程作用域，才能在里面合法地调用这些挂起函数，同时立即给调用方返回一个 `AsyncResult` 凭证（取餐器）。
3. **完美契合 Deferred 缓存池模型**
   在 `getOrPut` 中，我们需要向 `deferredInstances` 字典放入一个 `Deferred` 占位符。`scope.async { provider() }` 不仅把耗时初始化任务扔到了后台，而且**它自身的返回值正是一个 `Deferred`！** 这个设计堪称完美：第一个到达的请求直接用 `async` 开启后台组装并留下 `Deferred` 凭证，后续所有并发请求到达时，只需拿走同一个 `Deferred` 并调用 `await()` 即可，兵不血刃地解决了**防并发多次初始化**的世界级难题。

**模块通信设计总结：**
1. **零反射（Zero Reflection）**：依靠 KSP 生成的 `() -> Any` 工厂函数创建对象，性能达到原生的极致。
2. **懒加载（Lazy Loading）**：App 启动时只注册轻量级的 Map，绝不提前初始化任何实现类，大大降低了启动耗时。
3. **并发安全（Concurrency Safe）**：借助协程 `Mutex` 与 `CompletableDeferred`，即使在极端并发场景下，同一个模块服务也严格保证只被实例化一次。

## 二、 页面间双向通信源码解析

在原生开发中，页面回传数据通常使用 `startActivityForResult`，流程割裂且容易遇到生命周期问题。
天衢 利用 Kotlin 协程强大的**挂起(suspend)**与**恢复(resume)**机制，实现了犹如调用普通方法一样简单的页面双向通信：

```kotlin
// 期望的调用方式：
val result = navigator.navigateTo("app://SelectUserPage").await()
println("选中的用户是: $result")
```

### 1. 引擎层的调度 `Navigator.kt`

当我们在页面 A 调用 `navigateTo` 跳转页面 B 时，这个 `DeferredAsyncResult` 是如何流转的？

**步骤 A: 创建并传递 Result (Navigator#navigateTo)**
```kotlin
fun navigateTo(url: String, extra: Any? = null): AsyncResult<Any?> {
    // 1. 创建一个新的"承诺盒子" Result
    val result = DeferredAsyncResult<Any?>()
    
    // 2. 将盒子传递给实际处理路由导航的内部方法
    val success = processNavigation(url, extra, NavigationAction.PUSH, result)
    
    if (!success) {
        // 如果页面未找到或跳转失败，为了防止调用方死锁(永远挂起)，立刻用 null 完结盒子
        result.complete(null)
    }
    
    // 3. 将盒子返还给调用方 (页面A)。页面A立刻调用 await() 把自己挂起。
    return result
}
```

**步骤 B: StackEntry 妥善保管 Result (Navigator#processNavigation)**
```kotlin
private fun processNavigation(url: String, extra: Any?, action: NavigationAction, result: DeferredAsyncResult<Any?>?): Boolean {
    // ... 解析节点 ...
    
    // 创建目标页面 B 的导航栈节点时，将这个盒子 result 存入 B 的 StackEntry 中！
    val entry = StackEntry(matchedNode, context, result = result)
    
    // 压栈渲染
    backStack.add(entry)
}
```
**注意关键逻辑**：这个 `result` 盒子是由 页面A 创建的，但它被**存放在了页面B的 `StackEntry` 中**。

**步骤 C: 页面 B 销毁并回传数据 (Navigator#popBackStack)**
当用户在页面 B 选择了数据，并点击了返回按钮，会调用 `popBackStack(数据)`。

```kotlin
/**
 * @param result 返回给上一级页面的结果数据。被协程挂起等待的地方会收到该返回值。
 */
fun popBackStack(result: Any? = null) {
    if (backStack.size > 1) {
        // 1. 弹出栈顶页面 (即页面 B)
        val entry = backStack.removeLast()
        
        // 2. 🔥 核心：从页面 B 的节点中取出那个等待中的盒子，把数据塞进去！
        // 一旦调用 complete，原本挂起在 navigateTo(...).await() 的页面A 协程就会立刻苏醒，并拿到 result 数据！
        entry.result?.complete(result)
        
        // ... 通知重新组合渲染 ...
    }
}
```

## 三、 协程与回调双支持架构设计 (Dual-Paradigm Architecture)

作为一个底层框架，不能强制所有调用方都必须在协程环境中运行（比如有些旧代码或者非挂起函数的作用域）。因此，天衢 框架在**模块通信异步获取**和**页面双向通信**上，巧妙地设计了“既支持协程、又支持回调”的双范式架构。

### 1. 核心接口契约：`AsyncResult<T>`

框架抽象出了一个统一的结果获取接口，它同时提供了两种获取数据的方式：

```kotlin
interface AsyncResult<T> {
    /** 方式一：面向现代协程开发者，提供挂起函数 */
    suspend fun await(): T

    /** 方式二：面向传统回调开发者，提供监听器注册 */
    fun onResult(callback: (T) -> Unit)
}
```

- **模块异步获取**：`getServiceAsync<T>(...)` 返回的是 `AsyncResult<T>`。
- **页面路由跳转**：`navigateTo(...)` 返回的也是 `AsyncResult<Any?>`。

调用方可以根据自己的开发环境或习惯自由选择：

**场景 1：模块获取服务（ServiceLocator 的三态支持）**

对于服务获取，框架的设计更加完备，同时支持**同步**和**异步（协程+回调双范式）**：

```kotlin
// 1. 同步环境调用：直接获取，如果没初始化会立刻初始化并返回
val userService = ServiceManager.getService<UserService>()

// 2. 协程环境异步调用：挂起当前协程，等待并发环境下的服务安全初始化完成
lifecycleScope.launch {
    val userService = ServiceManager.awaitService<UserService>()
    // 底层等价于 resolver.getServiceAsync(...).await()
}

// 3. 传统回调异步调用：注册监听，如果服务正在后台初始化，完毕后触发回调
// (注: ServiceManager facade 屏蔽了直接调用，但在 Resolver 接口层面提供此能力)
resolver.getServiceAsync(UserService::class).onResult { userService ->
    println("服务初始化完毕并拿到了实例: $userService")
}
```

**场景 2：页面路由跳转带返回值（Navigator）**
```kotlin
// 协程环境调用：
lifecycleScope.launch {
    val result = navigator.navigateTo("app://user").await()
}

// 非协程环境调用（传统 Callback）：
navigator.navigateTo("app://user").onResult { result ->
    println("从用户页面拿到了返回值: $result")
}
```

### 2. 适配器实现模型 `DeferredAsyncResult.kt`

为了让这一套 API 底层只需要一套代码就能运作，框架内部通过**适配器模式（Adapter Pattern）**，将底层的 `CompletableDeferred` 包装成了对外的 `AsyncResult`。实现了同一套数据源驱动两种不同的接收方式。
为什么使用`CompletableDeferred`，而不是`async await`，`async`是自动调用完成方法，我们更多的是点击按钮才调完成方法，必须用`CompletableDeferred`。

```kotlin
class DeferredAsyncResult<T> : AsyncResult<T> {
    // 协程的"承诺"对象，代表一个未来的值
    private val deferred = CompletableDeferred<T>()
    
    // 保存普通回调函数的引用
    private var callback: ((T) -> Unit)? = null

    /**
     * 当数据准备好时（即目标页面 pop 回传数据，或模块异步初始化完毕时）被调用
     */
    fun complete(value: T) {
        // 1. 让挂起的 await() 立即恢复，并得到 value
        deferred.complete(value)
        // 2. 同时触发普通回调接口
        callback?.invoke(value)
    }

    /**
     * 协程挂起等待结果（直接透传给 Deferred）
     */
    override suspend fun await(): T {
        return try {
            deferred.await()
        } catch (e: kotlinx.coroutines.CancellationException) {
            throw e
        }
    }

    /**
     * 给非协程环境提供的回调监听
     */
    override fun onResult(callback: (T) -> Unit) {
        this.callback = callback
        
        // 关键保护：如果注册监听时，数据早就回来了（比如从缓存中秒回），
        // 必须直接调用一次 callback，防止"先完结、后监听"导致的数据丢失。
        if (deferred.isCompleted) {
            callback.invoke(deferred.getCompleted())
        }
    }
}
```

**双向通信设计总结：**
1. **双范式无缝兼容**：通过 `AsyncResult` 接口和 `DeferredAsyncResult` 适配器，底层使用同一个 `CompletableDeferred`，同时满足了现代协程和传统 Callback 两种开发习惯，不需要写两套代码。
2. **控制反转**：目标页面（B）的 `StackEntry` 在创建时，就被强行塞入了一个由调用方（A）创建的 `DeferredAsyncResult` 实例。
3. **生命周期完美闭环**：当目标页面出栈时，框架在底层统一拦截，触发 `deferred.complete()`，完美确保数据被回传且不会遗漏。即使目标页面是按系统的返回键（没有手动传 result），也会调用 `popBackStack(null)`，自动完结协程或触发 null 回调，彻底杜绝了**协程泄漏与回调死锁**的问题。
