# 路由框架启动模式（LaunchMode）实现原理详解

在 天衢 路由框架中，我们实现了类似 Android 原生 Activity 的启动模式（LaunchMode），主要包括 `STANDARD`（标准模式）、`SINGLE_TOP`（栈顶复用）和 `SINGLE_TASK`（栈内复用）。

本文档将深入源码，逐行解析 `SINGLE_TOP` 和 `SINGLE_TASK` 的实现原理、设计意图以及它们是如何与 Compose 的状态管理机制协同工作的。

---

## 核心概念：什么是复用？

在 Compose 声明式 UI 中，“复用”一个页面意味着：
1. **不创建新的 `StackEntry` 实例**：避免内存分配和额外的生命周期开销。
2. **保留页面的 UI 状态**：例如列表的滚动位置、输入框的文字等（依赖 `SaveableStateHolder`）。
3. **更新页面的参数（Context）**：虽然页面实例没变，但可能传入了新的参数（例如从 `/user?id=1` 跳转到 `/user?id=2`），页面需要根据新参数重新渲染。

## 源码解析：PushStrategy.kt

启动模式的核心逻辑集中在 `router-runtime/src/commonMain/kotlin/shijing/tianqu/runtime/strategy/PushStrategy.kt` 文件中。当执行 `Navigator.push()` 或 `Navigator.navigateTo()` 时，会调用 `PushStrategy.execute()` 方法。

```kotlin
class PushStrategy : NavigationStrategy {
    override suspend fun execute(
        navigator: Navigator,
        entry: StackEntry, // 即将入栈的新页面节点
        action: NavigationAction,
        url: String
    ): Boolean {
        // 1. 获取即将入栈页面的启动模式配置
        val launchMode = entry.node.launchMode.name
        // 2. 获取当前导航器的回退栈（这是一个 Compose 的 SnapshotStateList）
        val backStack = navigator.backStack

        when (launchMode) {
            // ... 分支逻辑 ...
        }
        
        // 如果是 STANDARD 模式，或者没有命中复用条件，则直接将新页面压入栈顶
        backStack.add(entry)
        return false 
    }
}
```

### 1. SINGLE_TOP (栈顶复用)

**设计意图**：如果即将跳转的页面，恰好就是当前处于栈顶（用户正在看）的页面，那么就不需要再创建一个新的实例压在上面，而是直接复用当前的栈顶页面，并把新的参数传递给它。
**解决的问题**：防止用户快速连点同一个按钮导致打开多个相同的页面；或者在当前页面刷新数据时避免不必要的页面切换动画。

```kotlin
"SINGLE_TOP" -> {
    // 【条件判断】：栈不能为空，且栈顶页面的路径（path）与即将入栈的页面路径相同。
    // 例如：当前栈顶是 "/user_profile"，即将跳转的也是 "/user_profile"。
    if (backStack.isNotEmpty() && backStack.last().node.path == entry.node.path) {
        
        // 获取当前栈顶的页面实例
        val topEntry = backStack.last()
        
        // 【精准拦截】：判断新传入的参数（entry.context）和当前页面的参数（topEntry.context）是否完全一致。
        // 如果参数完全一样（例如连点按钮），则什么都不做，直接拦截。
        if (!topEntry.context.isSameParameters(entry.context)) {
            
            // 【核心复用逻辑】：参数不同时，将新页面的 context 赋值给栈顶页面。
            // 因为 topEntry.context 在 StackEntry 中是被 mutableStateOf 包裹的（或者在渲染时被读取），
            // 这里的赋值操作会触发 Compose 的重组（Recomposition）。
            // 页面 UI 会使用新的参数重新绘制，但页面实例（topEntry）并没有改变。
            topEntry.context = entry.context
            
            // 标记这是一个新的 Intent（类似于 Android 的 onNewIntent）。
            // 业务层可以通过监听这个标记来执行特定的刷新逻辑（如果需要的话）。
            topEntry.isNewIntent = true
        }
        
        // 发送导航成功的事件通知（例如通知全局的日志系统或埋点系统）。
        navigator.coroutineScope.launch { navigator.emitEventForStrategy(RouterEvent.Navigated(action, url)) }
        
        // 返回 true，告诉 Navigator 外部：“我已经处理完毕了，你不需要再执行默认的入栈（backStack.add）操作了”。
        return true 
    }
}
```

### 2. SINGLE_TASK (栈内复用)

**设计意图**：如果即将跳转的页面在整个回退栈中已经存在（不一定在栈顶），那么就把它上面的所有页面全部出栈（销毁），让这个页面重新回到栈顶，并复用它。
**解决的问题**：常用于应用的主页（如 MainActivity）、购物车页面等。确保全局只有一个实例，并且当用户回到该页面时，能清理掉中间产生的临时页面，保持栈的整洁。

```kotlin
"SINGLE_TASK" -> {
    // 【查找目标】：从栈顶向栈底反向查找，找到最后一个路径匹配的页面索引。
    // 假设栈是 [A, B, C, D]，我们要跳转到 B (SINGLE_TASK)。
    // indexOfLast 会找到 B 的索引（1）。
    val existingIndex = backStack.indexOfLast { it.node.path == entry.node.path }
    
    // 如果找到了（索引不为 -1），说明栈内存在该页面，触发复用逻辑。
    if (existingIndex != -1) {
        
        // 计算需要移除的页面数量。
        // 在 [A, B, C, D] 中，B 的索引是 1，lastIndex 是 3。
        // removeCount = 3 - 1 = 2。即需要移除 C 和 D 两个页面。
        val removeCount = backStack.lastIndex - existingIndex
        
        // 【动画策略控制】：极其关键的一步！
        // 将最后一次导航动作设置为 POP_UNTIL（一直出栈直到...）。
        // 为什么这么做？因为在 ScreenRouterRenderer 中，AnimatedContent 会根据 lastAction 来决定播放什么动画。
        // 如果不设置，系统可能会认为这是一个普通的 PUSH（入栈）操作，从而播放错误的“新页面从右侧滑入”的动画。
        // 设置为 POP_UNTIL 后，系统会播放“旧页面向右滑出，底层页面重见天日”的返回动画，符合用户的直觉。
        navigator.setLastActionForStrategy(NavigationAction.POP_UNTIL)
        
        // 【清理上层页面】：循环移除目标页面之上的所有页面。
        for (i in 0 until removeCount) {
            // 每次都从栈顶（lastIndex）移除元素。
            val removed = backStack.removeAt(backStack.lastIndex)
            // 如果被移除的页面有协程在等待它的返回结果（navigateWithResult），
            // 这里传入 null 结束等待，防止协程泄漏。
            removed.result?.complete(null)
            // 调用 dispose() 触发页面的销毁生命周期回调。
            removed.dispose()
        }
        
        // 【核心复用逻辑】：获取目标页面（此时它已经成为了新的栈顶）。
        val targetEntry = backStack[existingIndex]
        
        // 同样进行参数比对，如果参数不同，则更新 context 触发重组。
        if (!targetEntry.context.isSameParameters(entry.context)) {
            targetEntry.context = entry.context
            targetEntry.isNewIntent = true
        }
        
        // 发送导航成功事件。
        navigator.coroutineScope.launch { navigator.emitEventForStrategy(RouterEvent.Navigated(action, url)) }
        
        // 返回 true，拦截默认的入栈操作。
        return true
    }
}
```

## 深度解密：为什么 SINGLE_TASK 之前会丢失状态？

在之前的开发过程中，我们遇到了一个极其隐蔽且棘手的 Bug。

**Bug 现象描述**：
1. 第一次进入 `SINGLE_TASK` 页面（我们称之为页面 A）。此时页面 A 内部生成了一个随机时间戳并显示在屏幕上。
2. 从页面 A 跳转到一个普通的中间页（页面 B）。此时页面 A 被页面 B 覆盖，退到了后台。
3. 在页面 B 中点击按钮，**再次跳转到页面 A**（触发 `SINGLE_TASK` 的栈内复用逻辑）。
4. **异常发生**：页面 A 重新显示在屏幕上，但是它上面显示的时间戳**改变了**！这意味着页面 A 并没有被真正复用，而是被 Compose 彻底销毁并重新创建了一次。
5. **更诡异的是**：如果你继续重复步骤 2 和 3（再次跳转到 B，再返回 A），页面 A 的时间戳却**不再改变**了，复用突然又生效了。

为什么第一次复用会失败，而后续的复用却能成功？要彻底理解这个问题，我们需要深入到 Compose 的状态管理和动画系统的最底层。

### 1. 状态是如何被保存的？

在 `ScreenRouterRenderer.kt` 中，我们使用了 `SaveableStateHolder` 来保存那些被移出屏幕的页面的状态：

```kotlin
saveableStateHolder.SaveableStateProvider(entry.id) {
    // 页面内容渲染
    entry.node.composable(entry.context)
}
```

当页面 A 被页面 B 覆盖时，页面 A 的 UI 节点会从 Compose 的组件树中被移除（为了节省内存）。但是，因为我们用 `SaveableStateProvider` 包裹了它，并且传入了 `entry.id` 作为唯一标识，Compose 会把页面 A 内部所有使用 `rememberSaveable` 声明的状态（比如那个随机时间戳）打包存放在内存中。

只要 `entry.id` 不变，且我们没有主动调用 `saveableStateHolder.removeState(entry.id)`，这份状态就永远存在。

### 2. 触发复用时的状态变更

当我们在页面 B 触发跳转回页面 A 时，`PushStrategy.kt` 中的 `SINGLE_TASK` 逻辑被执行：

```kotlin
// 1. 移除页面 B
for (i in 0 until removeCount) {
    val removed = backStack.removeAt(backStack.lastIndex)
    removed.dispose()
}

// 2. 获取页面 A 的实例
val targetEntry = backStack[existingIndex]

// 3. 极其关键的一步：修改页面 A 的上下文参数
if (!targetEntry.context.isSameParameters(entry.context)) {
    targetEntry.context = entry.context // 触发重组！
    targetEntry.isNewIntent = true
}
```

请注意第 3 步。在 `StackEntry.kt` 中，`context` 是一个被 `mutableStateOf` 包裹的变量：

```kotlin
// StackEntry.kt
var context: RouterContext by mutableStateOf(_context)
```

当我们执行 `targetEntry.context = entry.context` 时，我们实际上是修改了一个 Compose 的状态变量。这会向 Compose 框架发送一个信号：“这个变量的值改变了，所有读取过这个变量的 UI 组件都需要重新执行（重组）”。

### 3. 灾难的发生：AnimatedContent 的误判

在 `ScreenRouterRenderer.kt` 的最外层，我们使用了 `AnimatedContent` 来处理页面切换的动画：

```kotlin
val currentScreenEntry = navigator.backStack.lastOrNull { it.node.type == supportedType }

AnimatedContent(
    targetState = currentScreenEntry, // 监听栈顶页面的变化
    modifier = Modifier.fillMaxSize(),
    transitionSpec = { ... }
) { entry ->
    // ...
}
```

当页面 B 被移除后，`navigator.backStack.lastOrNull` 的结果从页面 B 变回了页面 A。`AnimatedContent` 敏锐地察觉到了 `targetState` 的变化，准备执行从 B 到 A 的退场动画。

**但是，此时的页面 A 已经不是“原来”的页面 A 了。**

因为我们在 `PushStrategy` 中刚刚修改了 `A.context`。

`AnimatedContent` 内部维护着一个它曾经渲染过的节点的缓存。当它准备重新渲染页面 A 时，它会拿当前传入的 `targetState`（修改了 context 后的 A）去和它内部缓存的旧节点进行比对。

**如果没有显式指定 `contentKey`，`AnimatedContent` 默认使用对象本身（即 `StackEntry` 实例）作为比对的 Key。**

在 Compose 极其复杂的底层 Diffing（差异比对）算法中，当一个对象（页面 A）从屏幕外重新进入屏幕内，**并且**它的内部状态（`context`）在进入的瞬间发生了改变，Compose 的底层机制在第一次遇到这种情况时，产生了误判。

它认为：“虽然这个对象的引用没变，但它的内部状态发生了剧烈变化，为了保证 UI 的绝对正确性，我不能冒险复用之前的节点结构，我必须把旧的节点彻底销毁，然后根据这个新的状态重新创建一个全新的节点。”

一旦 Compose 决定“销毁并重建”，它就不会去 `SaveableStateHolder` 里通过 `entry.id` 寻找之前缓存的状态了。它会像渲染一个全新页面一样，重新执行页面 A 的代码，重新生成那个随机时间戳。这就是为什么第一次复用会失败的原因。

*(至于为什么后续的复用会成功，这涉及到 Compose 内部缓存预热和状态快照的复杂时序问题。在第一次重建后，Compose 内部的某些状态树可能达到了某种稳定态，使得后续的 Diffing 算法能够正确识别出对象的同一性。但这属于框架底层的未定义行为，我们不能依赖它。)*

### 4. 终极修复：强行介入复用判定

为了彻底解决这个问题，我们必须剥夺 `AnimatedContent` 默认的、不可靠的比对权力，强行告诉它什么是“同一个页面”。

我们在 `ScreenRouterRenderer.kt` 中加入了 `contentKey` 参数：

```kotlin
AnimatedContent(
    targetState = currentScreenEntry,
    modifier = Modifier.fillMaxSize(),
    
    // 【修复 Bug 的定海神针】：显式指定 contentKey
    contentKey = { it.id },
    
    transitionSpec = { ... }
) { entry ->
    // ...
}
```

`it.id` 是 `StackEntry` 在创建时生成的一个全局唯一的 UUID（例如 `entry_1_4567`），它永远不会改变。

加上这行代码后，当页面 A 再次回到栈顶时，`AnimatedContent` 的比对逻辑变成了这样：

1. 拿到新的 `targetState`（修改了 context 后的页面 A）。
2. 提取它的 `contentKey`，得到 `entry_1_4567`。
3. 去内部缓存里找，发现：“哎？我这里正好有一个 Key 也是 `entry_1_4567` 的节点缓存！”
4. **无论页面 A 里面的 `context` 怎么变，只要 Key 匹配，`AnimatedContent` 就会绝对信任这是一个需要被复用的旧页面。**
5. 它会完美地恢复之前的节点结构，并从 `SaveableStateHolder` 中取出保存的时间戳状态。
6. 随后，因为 `A.context` 发生了改变，Compose 会触发局部的重组（Recomposition），仅仅更新那些依赖了 `context` 的 UI 元素（比如屏幕上显示的 URL 参数），而不会触碰那个被 `rememberSaveable` 保护的时间戳。

至此，`SINGLE_TASK` 的状态丢失 Bug 被彻底根除，实现了完美的栈内复用。

### 总结

- `SINGLE_TOP` 和 `SINGLE_TASK` 的核心在于**拦截默认的入栈操作**，并**修改已有页面的 `context`** 来触发重组。
- `SINGLE_TASK` 还需要额外处理**出栈清理**和**动画方向（POP_UNTIL）**。
- Compose 的状态保留依赖于 `SaveableStateHolder` 和稳定的标识符。在 `AnimatedContent` 中显式提供 `contentKey = { it.id }` 是保证跨页面层级状态复用不丢失的定海神针。
