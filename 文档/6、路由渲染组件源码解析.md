# 路由渲染组件逐行源码深度解析

在 天衢 路由框架中，`RouterHost` 并不直接负责把页面画出来，而是采用了**策略模式**，将具体的渲染工作委托给了不同的渲染器。
本篇文档将深入解析两大核心渲染器：`ScreenRouterRenderer`（负责全屏页面切换与动画）和 `DialogRouterRenderer`（负责弹窗）。

---

## 1. ScreenRouterRenderer：全屏页面的幕后推手

`ScreenRouterRenderer` 的主要职责是：从导航栈中找到**最顶层**的 `SCREEN` 类型页面，并利用 Compose 的 `AnimatedContent` 为其加上进出场动画，同时还要小心翼翼地维护好页面的生命周期和状态。

### 逐行代码拆解与设计意图

```kotlin
class ScreenRouterRenderer : RouterRenderer {
    // 1. 声明当前 Renderer 支持处理哪种类型的路由。
    // 设计意图：策略模式的基础。RouterHost 遍历所有 renderer，只有匹配 SCREEN 的节点才会进入这里的逻辑。
    override val supportedType: RouteType = RouteType.SCREEN

    @OptIn(ExperimentalSharedTransitionApi::class)
    @Composable
    override fun Render(navigator: Navigator, saveableStateHolder: SaveableStateHolder) {
        
        // 2. 从 Navigator 的回退栈（backStack）中，找到最后一个（也就是栈顶）类型为 SCREEN 的页面。
        // 设计意图：为什么只找“最后一个”？因为对于全屏页面，用户只能看到最上面的一层。
        // 底下的页面不需要渲染，Compose 会自动将它们从组合树中移除以节省极大的内存。
        val currentScreenEntry = navigator.backStack.lastOrNull { it.node.type == supportedType }

        // 3. 如果栈里有全屏页面，才开始下面的渲染。
        if (currentScreenEntry != null) {
            
            // 4. SharedTransitionLayout：这是 Compose 最新提供的共享元素动画容器。
            // 设计意图：必须在最外层包裹这个 Layout，里面所有的页面跳转才能在彼此之间传递需要共享的 UI 元素（比如从列表页点击图片，图片平滑放大到详情页）。
            SharedTransitionLayout {
                
                // 5. CompositionLocalProvider 依赖注入：将当前的共享元素作用域（this@SharedTransitionLayout）注入到上下文中。
                // 设计意图：因为子页面（如详情页）的 Composable 函数并不知道自己外面套了 SharedTransitionLayout。
                // 通过 LocalSharedTransitionScope，子页面可以通过 `Modifier.sharedElement()` 随处调用共享动画，实现了容器和页面的极致解耦。
                CompositionLocalProvider(LocalSharedTransitionScope provides this@SharedTransitionLayout) {
                    
                    // 6. AnimatedContent：Compose 官方强大的页面切换动画容器。
                    // 只要传入的 targetState 发生变化，它就会自动触发新旧两个页面的转场动画。
                    AnimatedContent(
                        targetState = currentScreenEntry, // 监听栈顶页面的变化
                        modifier = Modifier.fillMaxSize(), // 占满全屏
                        
                        // 7. transitionSpec：极其关键的一步！动态决定使用什么动画来切换。
                        transitionSpec = {
                            // 8. 判断当前导航操作是不是“出栈”（返回上一页）操作。
                            val isPopAnim = navigator.lastAction == NavigationAction.POP ||
                                            navigator.lastAction == NavigationAction.POP_TO_ROOT ||
                                            navigator.lastAction == NavigationAction.POP_UNTIL
                            
                            // 9. 动画策略选择：这行代码非常巧妙！
                            // 如果是“入栈（Push）”，那就看目标页面（targetState）配置了什么动画。
                            // 如果是“退栈（Pop）”，那就看即将被销毁的页面（initialState）配置了什么动画。
                            // 设计意图：假设 A->B 的入栈动画是“向左滑入”。那么 B->A 的退栈动画必须也是由 B 决定的（即“向右滑出”），所以出栈时必须读取 initialState。
                            val strategy = if (isPopAnim) initialState.node.transition else targetState.node.transition

                            // 10. 将具体的动画组装逻辑委托给具体的 TransitionStrategy 接口实现类。
                            strategy.buildTransitionSpec(
                                scope = this, 
                                initial = initialState, 
                                target = targetState, 
                                isPop = isPopAnim
                            )
                        },
                        label = "route_transition"
                    ) { entry -> // 这个 entry 是 AnimatedContent 提供给你的当前正在执行动画的节点（可能有新旧两个节点同时在渲染）
                        
                        // 11. 将 AnimatedVisibilityScope 注入。这是很多精细化进出场动画（如 Modifier.animateEnterExit）必须要用到的环境。
                        CompositionLocalProvider(LocalAnimatedVisibilityScope provides this@AnimatedContent) {
                            
                            // 12. SaveableStateProvider：恢复与保存 Compose 页面状态的核心（非常关键！）。
                            // 根据 entry.id 为这个页面开辟一块专属的“状态保险箱”。
                            // 设计意图：由于使用了 AnimatedContent，当 B 压入栈顶时，A 页面会被从 UI 树中移除。
                            // 这里提供了一个包裹层。注意：它本身不保存任何具体数据！它只是提供了一个【命名空间】。
                            // 当在业务层里调用 `rememberSaveable { mutableStateOf(...) }` 时，
                            // 这个 Provider 会拦截你的数据，并把它存进属于 `entry.id` 的保险箱里。
                            // 等下次返回 A 页面时，Provider 会把保险箱打开，把数据还给 `rememberSaveable`。
                            saveableStateHolder.SaveableStateProvider(entry.id) {
                                
                                // 13. DisposableEffect 监听节点的上屏与下屏生命周期。
                                DisposableEffect(entry) {
                                    
                                    onDispose {
                                        // 14. onDispose：当这个 Compose 节点从屏幕上彻底被移除（并且切换动画也播放完了）时触发。
                                        // isEntryAlive 会检查这个页面是不是还存在于导航栈中。
                                        // 设计意图：如果是因为页面真的被 POP 销毁了，那就彻底清理掉它缓存在全局的 UI 状态（removeState），防止内存泄漏。
                                        // 假如是因为压入了新页面导致的 A 页面被“暂时”移除，isEntryAlive 会返回 true，此时不会清空状态，等待未来恢复。
                                        if (!navigator.isEntryAlive(entry)) {
                                            saveableStateHolder.removeState(entry.id)
                                        }
                                    }
                                }

                                // 15. 注入 ViewModel 和 StackEntry 上下文
                                // 为什么需要 LocalViewModelStoreOwner？因为 Compose 里的 `viewModel()` 函数底层会去这个 Owner 里面拿 ViewModel。
                                // 我们把 StackEntry（它内部实现了 ViewModelStoreOwner 接口）提供出去。
                                // 设计意图：实现了“页面级 ViewModel”的存活域！哪怕页面组件经历了重组、哪怕因为入栈被暂时移除屏幕，只要它还在后退栈（Navigator.backStack）中，它的 ViewModel 实例就不会被销毁！
                                CompositionLocalProvider(
                                    LocalViewModelStoreOwner provides entry,
                                    LocalStackEntry provides entry
                                ) {
                                    // 16. 真正执行由开发者编写的 Compose 页面代码。
                                    entry.node.composable(entry.context)
                                }
                            }
                        }
                    }
                }
            }
        }
    }
}
```

## 2. DialogRouterRenderer：弹窗页面的多层堆叠艺术

与全屏页面最大的不同是：**弹窗是不遮挡底层的，并且弹窗之间可以互相叠加！**
因此，对于 Dialog 类型，我们不能只找“栈顶”的那一个，而是要把栈里所有的 Dialog 全都画出来。

### 逐行代码拆解与设计意图

```kotlin
class DialogRouterRenderer : RouterRenderer {
    override val supportedType: RouteType = RouteType.DIALOG

    @Composable
    override fun Render(navigator: Navigator, saveableStateHolder: SaveableStateHolder) {
        
        // 1. filter：与全屏的 `lastOrNull` 完全不同！
        // 弹窗可能存在多层，比如在“警告弹窗”上又弹出了一个“确认删除弹窗”。
        // 设计意图：我们需要把导航栈里所有的 DIALOG 都挑出来，依次渲染，这样才能形成视觉上的堆叠效果。
        val dialogEntries = navigator.backStack.filter { it.node.type == supportedType }
        
        // 2. 遍历每一个 Dialog 节点
        dialogEntries.forEach { dialogEntry ->
            
            // 3. 使用 Compose 官方提供的 `Dialog` 组件作为物理容器。
            // 它底层通常会开启一个独立的 Window（在 Android 上是系统级的 Dialog 窗口）。
            Dialog(
                // 4. onDismissRequest：当用户点击弹窗外部的灰色遮罩，或者按系统返回键时触发。
                // 设计意图：将物理操作桥接到路由框架。当用户试图关闭弹窗时，直接调用 navigator.popBackStack() 把它弹出栈。
                onDismissRequest = {
                    navigator.popBackStack()
                },
                // 5. properties 设置弹窗的平台属性。
                // usePlatformDefaultWidth = false：取消掉官方默认的屏幕两边留白的限制，允许我们自由控制弹窗的大小，甚至是全屏弹窗。
                properties = DialogProperties(usePlatformDefaultWidth = false)
            ) {
                // 后续代码与 ScreenRouterRenderer 的第 12 行到 18 行一模一样！
                // 即使是弹窗，我们依然要：
                // 1. 为它维护 SaveableStateHolder，防止配置变更（如屏幕旋转）丢失状态。
                // 2. 监听 DisposableEffect 清理已被 POP 掉的弹窗状态避免内存泄漏。
                // 3. 注入 ViewModelStoreOwner，允许开发者在弹窗级别作用域内使用独立的 ViewModel！
                
                saveableStateHolder.SaveableStateProvider(dialogEntry.id) {
                    DisposableEffect(dialogEntry) {
                        onDispose {
                            if (!navigator.isEntryAlive(dialogEntry)) {
                                saveableStateHolder.removeState(dialogEntry.id)
                            }
                        }
                    }
                    
                    CompositionLocalProvider(
                        LocalViewModelStoreOwner provides dialogEntry,
                        LocalStackEntry provides dialogEntry
                    ) {
                        dialogEntry.node.composable(dialogEntry.context)
                    }
                }
            }
        }
    }
}
```

## 总结：渲染引擎的核心设计哲学

1. **组合优于继承 (Composition vs Inheritance)**：
   我们没有写一个巨大的 BaseActivity 或 BaseFragment 去处理动画和 ViewModel，而是巧妙地利用 Compose 的 `CompositionLocalProvider` 把能力（作用域、ViewModel 管理器、状态暂存区）**隐式地**“倾泻”给被包裹的页面函数。页面代码中毫无感知，非常干净。
   
2. **生命周期精准解绑与控制**：
   在 Android 原生开发中，UI 的销毁和内存的回收往往绑定在一起。但在 Compose 中，UI（Compose 树节点）的移除仅仅是为了不画它。真正的数据（State, ViewModel）应该由业务需求（路由栈中是否存在）来管理。
   这就是为什么要有 `DisposableEffect` 搭配 `navigator.isEntryAlive` 的设计：只要你还在栈里，哪怕你看不见了，我不回收你的数据；只有你真正从栈里 pop 走了，我才会毫不留情地清空你的状态和 ViewModel 实例。

这套渲染架构同时兼顾了**性能、灵活性和多平台的纯 Compose 兼容性**，是一套极具参考价值的现代 KMP 路由渲染模型方案！
