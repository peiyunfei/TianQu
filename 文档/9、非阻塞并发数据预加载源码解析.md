# 非阻塞并发数据预加载源码深度解析

## 1. 痛点：为什么要设计路由预加载？

在传统的 App 开发中，打开一个需要网络数据的页面，我们通常会经历以下“瀑布流（串行）”加载过程：
1. 用户点击跳转按钮。
2. 框架开始执行页面切换动画（耗时约 300ms）。
3. 动画结束，新页面进入生命周期（`onCreate` / `LaunchedEffect`）。
4. 页面或 ViewModel 发起网络请求（耗时约 500ms~2000ms）。
5. 数据返回，页面刷新。

**问题在于**：在第 2 步到第 3 步这数百毫秒的时间里，网络空闲，不仅浪费了宝贵的时间，还增加了用户看到“白屏”或“加载圈”的时长。

**天衢 的非阻塞预加载架构**，就是要打破这个串行流程，让**页面转场动画**与**网络数据请求**实现**并发（Concurrent）执行**！这样等动画播完时，数据往往已经回来了，实现了“秒开”的极致体验。

---

## 2. 核心架构设计

### 2.1 契约接口：`RoutePreloader`

首先，框架定义了一个预加载器的标准接口，要求接入方实现。

```kotlin
/**
 * 路由数据预加载器接口
 * 接入方可以实现该接口并在 Navigator 中注册，框架会在跳转时自动触发 preload 挂起函数。
 */
interface RoutePreloader<T> {
    suspend fun preload(context: RouterContext): T
}
```

设计精髓在于 `suspend` 关键字。框架不关心你是读数据库、读文件还是发起网络请求，只要它是非阻塞的协程挂起函数即可。

---

### 2.2 引擎层调度：`Navigator.kt` 逐行解析

当调用 `navigator.navigateTo("...")` 时，底层是如何将请求派发出去的？

在 `Navigator.kt` 的 `processNavigation` 核心逻辑中：

```kotlin
// 1. 声明一个 Deferred 变量，用来保存"未来的预加载数据"
var preloaderDeferred: Deferred<Any?>? = null

// 2. 从预加载器注册表中，通过当前匹配的路由 path 获取对应的 Preloader
val preloader = preloaders[matchedNode.path]

if (preloader != null) {
    // 3. 🔥核心亮点：使用 async 开启一个并发协程去请求数据！
    preloaderDeferred = coroutineScope.async {
        try {
            // 调用业务方实现的 preload 挂起函数
            preloader.preload(context)
        } catch (e: Exception) {
            // 4. 关键防崩溃设计：内部捕获异常并直接作为返回值
            e
        }
    }
}

// 5. 将这个还在拼命拉取数据的 preloaderDeferred 凭证，强行塞给即将展示的页面的 StackEntry
val entry = StackEntry(matchedNode, context, result, preloaderDeferred = preloaderDeferred)

// 6. 将页面入栈（此时 UI 框架开始播动画，而后台协程正在疯狂拉数据，完美并发！）
backStack.add(entry)
```

**代码为什么这么设计？**
- **为什么用 `async` 而不是 `launch`？**
  因为我们需要返回值。`async` 返回的 `Deferred` 完美充当了一个“取餐凭证”，让页面能够在稍后通过 `await()` 获取数据。
- **为什么要 `catch (e: Exception)` 并返回 `e`？**
  在 Kotlin 协程中，如果一个子协程抛出异常，会导致整个父 `coroutineScope` 被取消（崩溃级灾难）。通过内部捕获并将异常本身当作结果 `return` 出去，不仅保护了路由引擎，还能让 UI 层通过判断结果是不是 Exception 来展示“网络错误页”。

### 2.3 UI 层接收：`rememberPreloadData.kt` 逐行解析

引擎层把拿到的 `Deferred` 塞到了 `StackEntry` 里，那 UI 组件该如何把这个异步的凭证，转换成 Compose 能够响应的 `State` 呢？

```kotlin
/**
 * 在页面内获取预加载的数据
 * @return 带有状态的数据。如果还在加载则为 null，加载完成或异常则返回 Result 对象包裹。
 */
@Composable
inline fun <reified T> rememberPreloadData(): State<Result<T>?> {
    // 1. 获取当前页面绑定的 StackEntry
    val entry = LocalStackEntry.current
    
    // 2. 拿出引擎层在 navigateTo 时塞进来的取餐凭证
    val deferred = entry?.preloaderDeferred
    
    // 3. 定义一个可观察的 Compose 状态，初始值为 null（代表 Loading）
    val state = remember { mutableStateOf<Result<T>?>(null) }

    // 4. 启动一个与页面生命周期绑定的副作用协程
    LaunchedEffect(deferred) {
        if (deferred != null) {
            try {
                // 5. 挂起等待！如果数据早就拉回来了，这里会瞬间通过。
                // 如果还在拉，协程在这里暂停，但不会卡顿 UI（转场动画照样丝滑播放）。
                val data = deferred.await()
                
                // 6. 处理 Navigator 层捕获并传过来的异常
                if (data is Exception) {
                    state.value = Result.failure(data)
                } else {
                    @Suppress("UNCHECKED_CAST")
                    // 7. 成功，包装成 Result.success 通知 UI 刷新！
                    state.value = Result.success(data as T)
                }
            } catch (e: Exception) {
                // 兜底防崩溃
                state.value = Result.failure(e)
            }
        }
    }

    return state
}
```

## 3. 最佳实践：结合 `tianquViewModel` 的业务运用

有了底层设施，在业务代码中应该怎么用？结合项目中的 `DemoPreloadScreen` 来看看。

**第一步：实现你自己的 Preloader**
```kotlin
class UserDetailPreloader : RoutePreloader<UserDetailData> {
    override suspend fun preload(context: RouterContext): UserDetailData {
        // 在这里发起网络请求 (挂起函数)
        // 它可以和页面转场动画并发执行
        return fetchUserDetailFromApi() 
    }
}
```

**第二步：在 Compose 页面中完美结合 ViewModel**
```kotlin
@Router(path = "/demo_preload")
@Composable
fun DemoPreloadScreen(context: RouterContext) {
    // 1. 获取拥有完整生命周期和状态恢复能力的 ViewModel
    val viewModel = tianquViewModel<DemoPreloadViewModel>()
    
    // 2. 从框架拿预加载的数据状态 (State)
    val preloadedResult by rememberPreloadData<UserDetailData>()

    // 3. 🔥 架构分层：将拉取到的数据喂给 ViewModel，保持单向数据流！
    LaunchedEffect(preloadedResult) {
        preloadedResult?.let { result ->
            viewModel.onPreloadResult(result)
        }
    }

    // 4. UI 永远只观察 ViewModel 的状态
    when (val state = viewModel.uiState) {
        is Loading -> CircularProgressIndicator() // 如果预加载比动画慢，展示菊花
        is Success -> Text("加载成功！姓名: ${state.data.name}") // 动画播完可能直接显示这个！
        is Error -> Text("加载失败！错误信息: ${state.message}")
    }
}
```

### 总结

天衢 路由的并发预加载机制，将**UI 转场驱动机制**与**协程异步并发**完美融合。它利用 `async` 启动任务，通过 `StackEntry` 传递凭证，最后由 `LaunchedEffect` 将结果桥接到 Compose 的响应式 `State` 中，是一套优雅、防崩溃且极致提升用户体验的高阶架构设计。
