# 天衢 路由导航引擎与参数传递原理解析

在 天衢 路由框架中，KSP 负责编译期的工作，而运行时的核心大脑就是 `Navigator.kt`（导航器）和 `UrlParser.kt`（URL解析器）。
本文档将对这两个核心类进行深度的“逐块/逐行”源码解析，不仅说明每一行代码的作用，更会深入探讨**为什么这样设计**。

---

## 一、UrlParser 核心解析与实例演示

`UrlParser` 是一个单例工具类，负责处理深层链接（DeepLink）、提取查询参数（Query）和提取路径参数（Path Params）。
在 KMP (Kotlin Multiplatform) 架构中，由于无法使用 Android 原生的 `Uri.parse()`，我们纯手写了一套跨平台的 URL 解析逻辑。

### 核心场景演示
假设我们定义了一个目标路由节点：
*   **Template (原始路径)**: `/user/{id}`
*   **Regex (生成的正则)**: `^/user/([\w-]+)$`

当用户触发跳转时，传入了如下的完整深层链接（带有中文 URL 编码）：
👉 `app://tianqu.com/user/10086?name=%E5%BC%A0%E4%B8%89&age=18`

### 1. `parseUrl(url: String)`
**作用**：剥离 scheme 和 host，将完整的 URL 拆分为 `纯路径(Path)` 和 `查询参数(Query)` 两个部分。
```kotlin
fun parseUrl(url: String): Pair<String, String> {
    var processedUrl = url
    // 1. 如果包含 "://" 说明是深层链接
    if (processedUrl.contains("://")) {
        // 截取掉 "app://" -> "tianqu.com/user/10086?name=..."
        val uriWithoutScheme = processedUrl.substringAfter("://")
        // 找到第一个 "/" 并保留后面的内容，拼上 "/" -> "/user/10086?name=..."
        processedUrl = "/" + uriWithoutScheme.substringAfter("/")
    }

    // 2. 以 "?" 为界，最多分割为两部分
    val parts = processedUrl.split("?", limit = 2)
    val path = parts[0] // 结果："/user/10086"
    val query = if (parts.size > 1) parts[1] else "" // 结果："name=%E5%BC%A0%E4%B8%89&age=18"
    return Pair(path, query)
}
```
*   **设计思考**：移动端常常通过 Schema (如 `app://`) 从外部唤起 App。路由框架在匹配页面时，只关心具体的业务路径（`/user/...`），因此必须有一层剥离机制，让内外跳转逻辑归一化。

### 2. `parseQuery(query: String)`
**作用**：解析 `&` 连接的查询参数，并进行 URL 解码。
```kotlin
fun parseQuery(query: String): Map<String, List<String>> {
    // 省略部分代码...
    // 将 "name=%E5%BC%A0%E4%B8%89&age=18" 按 "&" 拆分
    query.split("&").forEach {
        val kv = it.split("=", limit = 2)
        val key = decode(kv[0]) // 解码 Key
        val value = if (kv.size > 1) decode(kv[1]) else "" // 解码 Value
        // 结果：mapOf("name" to listOf("张三"), "age" to listOf("18"))
        result.getOrPut(key) { mutableListOf() }.add(value)
    }
    return result
}
```
*   **设计思考**：为什么 Value 是 `List<String>` 而不是 `String`？因为 URL 规范中允许同名参数出现多次（例如 `?tag=a&tag=b`），使用 List 能保证数据不丢失。

### 3. 路由节点匹配：`findMatchedNode` (位于 Navigator.kt)
**作用**：在解析出纯路径（Path）后，框架需要在注册表中找到对应的路由节点（RouterNode）。
```kotlin
private fun findMatchedNode(path: String): RouterNode? {
    return routeRegistry.find { node ->
        Regex(node.regexPattern).matches(path)
    }
}
```
*   **设计思考**：为什么不用简单的字符串相等（`==`）去匹配？
    因为动态路由的路径是可变的（例如 `/user/10086` 和 `/user/10010` 都属于 `/user/{id}` 这个节点）。在 KSP 编译期，我们利用处理器将 `@Router` 上的路径自动转换成了真正的正则表达式存放在 `RouterNode.regexPattern` 中。

**【关于路由节点中 regexPattern 的深度解析】**

在 KSP 生成的 `RouterNode` 代码中，你经常会看到类似这样的属性赋值：
`regexPattern = "^/user/([\\w-]+)\${'$'}"`

1. **`regexPattern` 的核心作用是什么？**
   它的作用是将开发者的直观路径（如 `/user/{id}`）翻译成机器能够精确匹配并提取数据的规则。
   * **匹配路由**：当收到任意真实 URL（如 `/user/10086`）时，框架首先依靠它来判断这个 URL 是否属于当前节点。
   * **提取参数**：通过正则内的捕获组，它可以将 URL 里的动态值抠出来传递给目标页面。

2. **为什么要加上 `^` 和 `$` ？**
   在编译期生成的正则通常长这样：`^/typesafe_demo$`。
   * `^` 代表字符串开头。
   * `$` 代表字符串结尾。
   * **设计意图**：这是为了**强制全量精确匹配**。如果不加 `$`，那么当用户跳转 `/typesafe_demo/extra` 时，也会错误地部分命中 `/typesafe_demo` 这个节点，导致路由混乱和严重 Bug。

3. **代码里奇怪的 `${'$'}` 是什么意思？**
   这是一个典型的 Kotlin 代码生成器的语法细节。
   在 Kotlin 双引号字符串中，`$` 具有特殊意义，用于**变量插值**（如 `"$name"`）。如果我们直接写 `^/user/([\\w-]+)$`，Kotlin 编译器会试图去寻找一个名字为空的变量，从而引发语法错误。
   为了强行告诉编译器“这就是一个普通的文本美元字符，请原封不动地输出它”，标准解法是用花括号包裹成单字符表达式：`${'$'}`。**它在编译运行后，实际上就等于一个纯纯的 `$` 字符。**
   * **追问：换成其他特殊字符还需要这样写吗？**
   * **不需要**。如果在这里你使用 `@` 或 `#` 等符号，因为它们在 Kotlin 中没有插值含义，直接写 `"^/user/([\\w-]+)@"` 即可。但问题是，**正则标准死规定了只有 `$` 能代表“字符串结尾”**，你不能换成别的字符。这实际上是“正则的 `$` 标准”与“Kotlin 字符串的 `$` 插值”发生了**符号冲突**，所以不得不采用 `${'$'}` 来曲线救国。

4. **`([\w-]+)` 是干什么的？**
   括号 `(...)` 在正则中叫做**“捕获组（Capture Group）”**。
   * `\w` 匹配字母、数字、下划线。
   * `-` 匹配横杠（用于匹配 UUID 等带中划线的值）。
   * `+` 表示至少出现一次。
   这意味着它是一个万能容器，能匹配像 `123`、`abc_12`、`user-uuid` 这样的参数内容，并将这些内容捕获下来。

5. **把 `{id}` 换成 `{name}`，生成的正则也是一样的吗？**
   **是的！完全一样！**
   在 KSP 的处理器源码中，所有的占位符（无论是 `{id}`、`{name}` 还是 `{uuid}`）都会被统一的逻辑替换成 `([\w-]+)`。
   * **那框架在运行时怎么知道提取出来的值对应哪个参数名呢？**
   * 答案是：**按顺序对号入座**。框架在 `extractPathParams` 方法中，会先解析原始的 template 路径，按先后顺序提取出所有的变量名（如 `["id", "name"]`），然后再用正则去匹配真实 URL，按顺序提取出捕获组里的值（如 `["10086", "zhangsan"]`），最后把两个数组像拉链一样缝合（Zip）成 Map，完美还原参数映射表。

**【架构设计亮点：为什么放弃手写字符串解析，选择正则表达式作为动态路由引擎？】**

“既然正则可读性差、程序员记不住规则，你为什么非要在路由框架里引入正则，而不是自己手写一个字符串分割匹配（比如用 `split("/")` 算法）？”
这其实是路由框架的一个**核心架构亮点**，你可以用以下四个维度的递进逻辑来回答：

1. **开发者“零负担”（对业务层完全透明）**
   最核心的一点是：**使用框架的业务程序员，根本不需要手写任何一行正则！**
   业务开发者只需写非常直观的占位符路径：`@Router("/user/{id}")`。所有的正则转换逻辑（替换成 `^/user/([\w-]+)$`），全都是在 **KSP 编译期由框架底层自动完成的**。正则的复杂性被完全屏蔽了。
2. **拒绝“硬编码解析”（性能与健壮性碾压）**
   如果不依靠正则，匹配 `/user/10086/detail/zhangsan` 和模板 `/user/{id}/detail/{name}` 需要怎么做？你可能得：用 `split("/")` 切割数组 -> 判断长度是否一致 -> 循环遍历 -> 遇到 `{}` 跳过匹配 -> 不是 `{}` 的对比是否相等。这种手写的解析代码极其丑陋，且极易出现数组越界、多斜杠、空字符串等边界 Bug。
   而**正则表达式底层是高度优化的有限状态自动机 (DFA/NFA)**。只要预编译好，匹配速度极快且极其健壮。
3. **“一石二鸟”（匹配与参数提取一次搞定）**
   手写代码最大的痛点在于提取参数：匹配成功后，你还得记住哪个索引位置是变量，再回过头去原 URL 的相同索引抠值。
   而正则的**捕获组 `(...)` 天生就是干这个的**。`Regex.matchEntire()` 跑完后，不仅告诉你是不是这个路由，还顺手把 `10086` 和 `zhangsan` 塞进了 `groupValues` 数组里打包送给你。验证与提取，一行代码一次搞定，极其优雅。
4. **业界标准（行业普适性）**
   纵观业界，无论是前端大名鼎鼎的 `Vue Router`、`React Router`，还是后端的 `Spring Boot`、`Express.js`，它们底层的动态路由算法，**无一例外全是将模板字符串编译成正则表达式来执行的**。路由匹配本质上就是一个典型的“模式匹配”问题，把专业的事交给专业的引擎去做，才是做架构设计的基本原则。

**【框架具体是如何使用正则串联全流程的？】**
整个流程分为**编译期准备**和**运行期执行**两步，环环相扣：
*   **第一步：编译期翻译 (KSP)**
    开发者写下 `@Router("/user/{id}/detail/{name}")`。KSP 处理器使用 `replace` 算法，将所有形如 `{xxx}` 的部分无差别替换为 `([\w-]+)`，并在首尾加上边界符。生成并硬编码进注册表的字符串变为：`^/user/([\w-]+)/detail/([\w-]+)$`。
*   **第二步：运行期验证 (`Navigator.findMatchedNode`)**
    当用户发起跳转请求 `/user/10086/detail/zhangsan` 时，框架拿着这个真实的 URL，遍历注册表。一旦 `Regex("^/user/([\w-]+)/detail/([\w-]+)$").matches(url)` 返回 true，说明找到了目标节点。
*   **第三步：运行期提取与缝合 (`UrlParser.extractPathParams`)**
    找到节点后，框架会先用正则把原始模板 `/user/{id}/detail/{name}` 里的变量名全捞出来形成数组：`["id", "name"]`。
    然后再用编译好的大正则对真实 URL 提取捕获组结果形成数组：`["10086", "zhangsan"]`。
    最后，利用 Kotlin 的 `forEachIndexed`，按先后顺序将这两个数组“像拉链一样”缝合起来，得到最终的字典：`Map("id" to "10086", "name" to "zhangsan")`，并透传给最终的业务 Compose 页面。

**【实战踩坑：如果跳转路径与注解模板不匹配会怎样？】**

**问题场景**：假设注解定义为 `@Router("/user/{id}/detail/{name}")`，但在实际跳转时，开发者手误写成了 `navigator.navigateTo("/user/10086/zhangsan")`（漏掉了中间的 `/detail`），这时会发生什么？

**框架处理流程**：
1. **正则匹配失败**：框架拿着生成的正则 `^/user/([\w-]+)/detail/([\w-]+)$` 去匹配实际传入的 `/user/10086/zhangsan`。由于实际路径中缺失了硬编码的 `/detail` 文本段，正则匹配直接返回 `false`。
2. **节点查找为空**：`findMatchedNode` 方法遍历完整个路由表都没有找到匹配项，最终返回 `null`。
3. **触发降级机制 (404)**：在 `Navigator.processNavigation` 的核心逻辑中，发现 `matchedNode == null`，此时框架会认为这是一个**无效路由（404 Not Found）**。
4. **全局处理**：框架会尝试调用外部注入的 `routerHandler?.handleNotFound(url)`。如果你配置了全局的 404 降级页面，就会跳转到 404 页面；如果没有配置，框架会通过 `routeEvents` 发射一个 `RouterEvent.NotFound` 事件，并直接中断本次跳转（返回 `false`）。不会引发 App 崩溃，但页面也不会有任何响应。

**总结**：动态路由的占位符仅仅是替换变量，**路径中的静态常量部分（如 `/detail`）必须严格对齐**，否则就会触发 404 降级逻辑。

### 4. 提取路径参数：`extractPathParams` (位于 UrlParser.kt)
**作用**：在通过 `findMatchedNode` 找到目标节点后，利用正则捕获组，将路径上的动态占位符（如 `{id}`）提取出来。
```kotlin
fun extractPathParams(template: String, regexPattern: String, actualPath: String): Map<String, String> {
    // 1. 从模板 "/user/{id}" 中提取出变量名集合 ["id"]
    val paramNames = Regex("\\{([^/]+)\\}").findAll(template).map { it.groupValues[1] }.toList()
    
    // 2. 用 "/user/10086" 去匹配正则 "^/user/([\w-]+)$"
    val matchResult = Regex(regexPattern).matchEntire(actualPath)
    if (matchResult != null) {
        // matchResult.groupValues 是 ["/user/10086", "10086"]
        // drop(1) 去掉第 0 个全匹配，剩下捕获组 ["10086"]
        val values = matchResult.groupValues.drop(1)
        
        // 3. 将变量名 "id" 和 捕获值 "10086" 一一对应存入 Map
        paramNames.forEachIndexed { index, name ->
            result[name] = decode(values[index])
        }
    }
    // 最终返回 mapOf("id" to "10086")
}
```
*   **设计思考**：这是动态路由的灵魂。正则捕获组（Group）的天生特性非常适合这种按位置提取变量的需求。

## 三、高级传参机制原理解析：内存对象与强类型传参

除了通过 URL 的 Path（路径参数）和 Query（查询参数）传递简单的字符串外，现代路由框架必须能够应对复杂的业务场景。天衢 提供了两种高级传参方案：**内存级复杂大对象传递（Extra）** 和 **基于 Kotlinx.serialization 的强类型安全传参**。

### 1. 内存级复杂大对象传递 (`extra: Any?`)
**使用场景**：从列表页跳转到详情页时，直接把列表页已经拉取到的超大 JSON 列表、甚至是 Bitmap 图片、回调函数 (Callback) 传给下一个页面。

**实现原理**：
在 `Navigator.navigateTo(url, extra)` 和 `RouterContext(url, ..., extra)` 中，`extra` 参数的类型被定义为 `Any?`。
它**完全不参与 URL 的拼接和解析**。它的原理极其简单且粗暴：**内存引用传递（指针拷贝）**。
当新页面入栈时，框架生成了一个 `StackEntry` 压入 `backStack`，这个 `extra` 对象就安静地躺在这个 Entry 的内存里。目标页面可以直接从 `context.extra` 中取出它强制转换后使用。

**设计思考（优缺点）**：
*   **优点**：**极致的性能，毫无限制**。不需要经过任何序列化/反序列化，也不关心对象里装的是什么（甚至可以装 Android 的 Context 或者函数引用），传输耗时为 0。
*   **缺点**：**致命的“脆弱性”**。它只存在于当前进程的内存中。一旦发生**进程被系统杀掉后重建（Process Death & Restoration）**，或者用户通过外部浏览器唤起 App（DeepLink跳转），这个 `extra` 就会变成 `null`。因此，它只能用于强依赖当前进程上下文的端内临时跳转。

### 2. 强类型安全传参 (`TypeSafeArgs`)
**使用场景**：规范化页面间的数据契约。避免手动拼接 URL 导致参数名写错、类型转换崩溃（比如把 String 强转 Int 报错），同时要求完美支持跨进程和外部唤起。

**实现原理**：
在 `TypeSafeArgs.kt` 中，框架利用了 `kotlinx.serialization` 提供了一组内联扩展函数：
```kotlin
inline fun <reified T : Any> Navigator.navigateArgs(path: String, args: T) {
    // 1. 将数据类序列化为 JSON 字符串
    val jsonString = RouterJson.encodeToString(serializer<T>(), args)
    // 2. 将 JSON 字符串进行 UrlSafe Base64 编码
    val base64Payload = Base64.UrlSafe.encode(jsonString.encodeToByteArray())
    // 3. 将编码后的长字符串作为特殊查询参数 __typed_args 追加到 URL 后面
    val url = "$path?__typed_args=$base64Payload"
    navigateTo(url)
}
```
接收端调用 `context.getTypedArgs<T>()` 时，框架进行**反向操作**：从 URL 提取出 Base64 字符串 -> 解码为 JSON -> 反序列化为原始的数据类对象。

**设计思考**：
1. **为什么要过一道 Base64 编码，而不直接把 JSON 拼在 URL 后面？**
   JSON 字符串中含有大量的双引号 `"`、大括号 `{}`、逗号 `,`。如果直接拼在 URL 的 `?` 后面，一方面会破坏 URL 的标准格式引发解析异常，另一方面如果 JSON 数据里恰好包含 `&` 或 `=`，会直接扰乱路由原本的查询参数解析器。采用 **UrlSafe Base64** 编码，将整个复杂对象压缩成一串纯粹的、安全的“字母数字乱码”，完美隔离了业务数据与底层路由协议。
2. **真正的类型安全（Type-Safe）与内联具体化（inline & reified）**：
   框架中的 `navigateArgs` 和 `getTypedArgs` 方法都使用了 `inline` 和 `reified T`。为什么要这样设计？
   * **打破泛型擦除（Type Erasure）**：在普通的 Kotlin/Java 泛型中，类型 `T` 在运行期会被擦除为 `Object`。但 `kotlinx.serialization` 的 `serializer<T>()` 必须在**运行期**明确知道待序列化的真实 Class 类型。
   * **`reified`（具体化泛型）**：允许我们在函数体内像使用普通类一样直接使用泛型 `T`。
   * **`inline`（内联展开）**：`reified` 必须和 `inline` 配合使用。因为 `inline` 会让编译器在**调用点**（Call Site）直接将函数体的字节码复制粘贴过去。此时，编译器是完全知道开发者传入的真实类型（比如 `UserArgs`）的，从而在编译期就将 `T` 替换成了具体的类，完美解决了泛型擦除问题。
   这不仅实现了代码的极其简洁（调用方无需传入 `UserArgs::class`），还在编译期完成了严格的类型约束，从根本上杜绝了隐式的类型转换异常（ClassCastException）。
3. **完美兼容 DeepLink 与进程恢复**：
   由于序列化后的数据最终**完全融入了 URL 本身**，它就享受了 URL 的一切特权。无论你是生成一个二维码让用户扫码跳进来，还是 App 在后台被系统杀了重新恢复，只要这串长长的 URL 还在，页面就能完美还原出当初传过来的那个复杂数据类。这是 `extra` 内存传参绝对做不到的。

**总结**：`extra` 传参是“走捷径”，牺牲持久化换取性能与灵活性；而 `TypeSafeArgs` 是“修正道”，牺牲了一丢丢 JSON 序列化的微小耗时，换来了完美的健壮性、类型安全和全场景支持。在现代 App 架构中，应尽可能首选强类型安全传参。

## 三、Navigator 核心引擎源码解析

`Navigator` 相当于 Android 中的 `FragmentManager` 或 `NavController`，但它是完全基于 Compose 状态驱动的。

### 1. Navigator 的获取方式：CompositionLocal 与协程上下文

在 Compose 树中，如果每个子组件都需要跳转页面，把 `Navigator` 作为参数一层层往下传（Prop Drilling）是非常痛苦的。框架利用了 Compose 的 `CompositionLocal` 机制来解决这个问题：

在 `RouterHost.kt` 中定义了：
```kotlin
val LocalNavigator = compositionLocalOf<Navigator> {
    error("No Navigator provided. Make sure to wrap your content with RouterHost.")
}
```
在根节点 `RouterHost` 渲染时，通过 `CompositionLocalProvider(LocalNavigator provides navigator)` 将导航器注入到整个组件树的上下文中。
**业务侧使用**：在任何深层级的 Composable 函数中，只需调用 `val navigator = LocalNavigator.current` 即可直接获取到全局（或当前嵌套层级）的导航器实例，极大地解耦了 UI 组件与路由引擎。

除了 `CompositionLocal`，框架还提供了另一种获取方式：
```kotlin
class Navigator(...) : AbstractCoroutineContextElement(Navigator) {
    companion object Key : CoroutineContext.Key<Navigator>
    // ...
}
```
*   **设计思考**：为什么继承 `AbstractCoroutineContextElement`？
    这是一种非常高级的协程编程技巧！通过让 Navigator 成为协程上下文的一部分，我们可以像这样在任何协程作用域内随意获取它：`coroutineContext[Navigator]`。这彻底摆脱了传统框架中需要不断传递 Context 或使用全局静态单例的丑陋做法，使架构更加函数式和模块化。

### 2. 状态驱动的返回栈
```kotlin
val backStack = mutableStateListOf<StackEntry>()
private val savedStateStacks = mutableMapOf<String, List<StackEntry>>()
var lastAction by mutableStateOf(NavigationAction.IDLE)
    private set
```
*   **代码解读与设计思考**：
    *   `mutableStateListOf`：这是 Compose 独有的状态列表。只要往这个列表 `add` 或 `remove` 元素，Compose 就会**自动触发 Recomposition（重组）**，从而更新 UI。这就是为什么 Navigator 只需要操作数据，界面就能自动切换的原因。
    *   `savedStateStacks`：这是为了支持**“多 Tab 切换”**设计的。当你从“首页 Tab”切到“我的 Tab”时，如果不保存状态，首页的滚动位置和历史记录就丢了。这个 Map 就是把被隐藏的 Tab 的回退栈（List）完整打包存起来，等切回来时再原封不动地放回 `backStack` 中。
    *   `lastAction`：使用 `mutableStateOf`。记录刚才是 PUSH 还是 POP，路由容器（RouterHost）会监听这个状态，决定此时页面切换该用左滑动画还是右滑退出动画。

### 3. 极简的事件总线 (Event Bus)
```kotlin
private val _routeEvents = MutableSimpleSharedFlow<RouterEvent>()
val routeEvents: SimpleSharedFlow<RouterEvent> = _routeEvents
```
*   **设计思考**：跳转成功了没？拦截器生效了吗？页面找不到（404）怎么办？外部系统（比如全局弹窗、数据上报层）需要知道路由动作。与其到处写回调 (Callback)，不如暴露一个冷流（基于挂起的 SharedFlow），让外部按需 `collect`。这也是响应式编程的最佳实践。

### 4. 核心路由枢纽：`processNavigation`
这是整个框架最复杂、最关键的函数，所有的跳转最终都会走到这里。
```kotlin
private suspend fun processNavigation(url: String, extra: Any?, action: NavigationAction, result: DeferredAsyncResult<Any?>? = null, saveState: Boolean = false, restoreState: Boolean = false): Boolean {
    // 1. 第一步：解析 URL，尝试寻找匹配的节点，生成上下文 Context
    var (matchedNode, context) = prepareContext(url, extra)

    // 2. 第二步：过安检！执行全局路由守卫（例如：检查是否登录，或者动态下载某些还没加载的模块）
    if (!GuardProcessor.processGuards(context, guards)) {
        return false // 被拦截器叫停，直接阻断跳转
    }
    
    // 3. 第三步：重新匹配。如果在守卫里下载了新模块（注入了新路由），之前没找到的节点现在可能找到了！
    if (matchedNode == null) {
        val (newNode, newContext) = prepareContext(url, extra)
        matchedNode = newNode
        context = newContext
    }

    // 4. 第四步：兜底降级。真找不到，就交给 routerHandler 外部处理器（比如打开一个全局 404 页面）
    if (matchedNode == null) {
        // 省略降级逻辑...发送 NotFound 事件...
        return false
    }

    // 5. 第五步：处理多返回栈状态保存与恢复（切换 Tab）
    if (saveState && backStack.isNotEmpty()) {
        // 把现在的栈打包存进 Map
        savedStateStacks[backStack.first().node.path] = backStack.toList()
        backStack.clear()
    }
    if (restoreState) {
        // 从 Map 中把之前存的栈掏出来，覆盖当前栈，跳转结束！
        val restored = savedStateStacks.remove(matchedNode.path)
        if (!restored.isNullOrEmpty()) {
            backStack.addAll(restored)
            return true
        }
    }

    // 6. 第六步：触发异步预加载（网络请求优化）
    var preloaderDeferred: Deferred<Any?>? = null
    val preloader = preloaders[matchedNode.path]
    if (preloader != null) {
        // 不阻塞当前线程，后台开协程去预加载数据（比如在跳往商品详情页的瞬间，并发发起网络请求）
        preloaderDeferred = coroutineScope.async { preloader.preload(context) }
    }

    // 7. 第七步：入栈！将页面封装为 StackEntry 压入 Compose 状态栈
    val entry = StackEntry(matchedNode, context, result, preloaderDeferred = preloaderDeferred)
    when (action) {
        NavigationAction.PUSH -> backStack.add(entry)
        NavigationAction.REPLACE -> {
            backStack.removeAt(backStack.lastIndex).dispose()
            backStack.add(entry)
        }
        NavigationAction.CLEAR_AND_PUSH -> {
            backStack.clear()
            backStack.add(entry)
        }
    }
    
    return true
}
```

### 5. 高级跳转 API 设计

为了应对不同的业务需求，Navigator 封装了多种风格的 API：

**常规跳转：**
```kotlin
fun navigateTo(url: String, extra: Any? = null) {
    coroutineScope.launch { processNavigation(...) }
}
```
*   **设计思考**：Fire and forget（发射后不管）。不用等它走完，适合绝大多数普通的点击跳转场景。

**挂起跳转（带返回结果）：**
```kotlin
suspend fun awaitNavigateForResult(url: String, extra: Any? = null): Any? {
    return navigateWithResult(url, extra).await()
}
```
*   **设计思考**：这是一个非常强大的功能！假设我们从 A 页面跳到 B 页面（选人列表），选完人回来。
    在传统的 Android 开发中，你需要用 `startActivityForResult` 甚至写一大堆恶心的回调地狱，或者用 EventBus 传来传去。
    现在，利用 Kotlin 协程的挂起特性，我们可以写出同步风格的代码：
    ```kotlin
    // 在 A 页面
    val selectedUser = navigator.awaitNavigateForResult("/selectUser")
    if (selectedUser != null) { ... }
    ```
    当调用此方法时，当前协程被**挂起（暂停执行）**。直到 B 页面调用了 `popBackStack(result)`，将结果传回并**唤醒（恢复执行）**协程，代码才会继续往下走。这极大地提升了业务代码的阅读性和维护性。

**返回动作：**
```kotlin
fun popBackStack(result: Any? = null) {
    if (backStack.size > 1) {
        lastAction = NavigationAction.POP
        val entry = backStack.removeAt(backStack.lastIndex) // 从栈中弹出
        entry.dispose() // 释放页面的内存（如 ViewModel）
        
        // 关键逻辑：如果上一级页面正在 awaitNavigateForResult 挂起等待，把结果塞给它，唤醒它！
        entry.result?.complete(result)
    } else {
        // 到底了，如果是嵌套路由，就交给外层的父 Navigator 去响应物理返回键
        parent?.popBackStack(result)
    }
}
```
*   **设计思考**：`result?.complete(result)` 完美闭环了上面提到的等待机制。`parent?.popBackStack` 则是为了解决“首页里面嵌套了一个带导航的 Tab，用户点物理返回键时，到底是切 Tab 还是退出 App”这种经典的嵌套路由分发问题。
