# StackEntry 与 TianquViewModel 逐行源码深度解析

在 天衢 路由框架中，如何优雅地管理页面的生命周期和状态，是决定框架稳健性的关键。本篇文档将深入拆解 `StackEntry.kt` 与 `TianquViewModel.kt` 的源码，为你解释它们每一行代码背后的设计意图，以及如果不使用这些设计会引发怎样灾难性的后果。

---

## 1. 必读：前置核心知识补充

如果你在阅读后续源码时感到困惑，很可能是因为对 Compose 和 ViewModel 的底层运行机制不够熟悉。请先理解以下 4 个核心概念，它们是整个路由框架设计的基石：

### 概念一：ViewModel 到底存在哪里？（ViewModelStore 与 Owner）
*   **`ViewModelStore`**：你可以把它简单理解为一个**大 Map**（`HashMap<String, ViewModel>`）。ViewModel 实例就存放在这个 Map 里。
*   **`ViewModelStoreOwner`**：这是一个接口，意思是**“我是这个 Map 的主人”**。谁实现了这个接口，谁就负责保管这个 Map。
*   在传统的 Android 开发中，`Activity` 和 `Fragment` 默认实现了这个接口，所以 ViewModel 跟着 Activity/Fragment 存活。

### 概念二：Compose 官方的 `viewModel()` 函数是怎么工作的？
当你在 Compose 页面里写下 `val vm: MyViewModel = viewModel()` 时，系统并不是直接 `new` 了一个对象，而是执行了以下“寻宝”过程：
1.  Compose 会顺着当前的 UI 树**往上爬**，寻找离自己最近的 `ViewModelStoreOwner`。
2.  找到 Owner 后，向 Owner 索要那个大 Map（`ViewModelStore`）。
3.  去 Map 里查：有没有 `MyViewModel`？
    *   如果有，直接返回给你（这就是为什么 ViewModel 能保持状态不丢失）。
    *   如果没有，就创建一个新的，**存进 Map 里**，再返回给你。

### 概念三：`CompositionLocal` 是什么魔法？（隐式依赖注入）
在 Compose 中，如果父组件想把一个数据传给很深层级的子组件，一层层传参数会累死人。
`CompositionLocal` 就像是一个**“隐形的作用域广播”**。
当我们在父组件写下：
```kotlin
CompositionLocalProvider(LocalViewModelStoreOwner provides myOwner) {
    // 这里的子组件
}
```
就相当于在广播：“在这个大括号里的所有子组件，如果你们要找 `ViewModelStoreOwner`，别往上爬去找 Activity 了，**直接用我提供的 `myOwner`！**”

### 概念四：UI 树的销毁 vs 内存对象的存活
*   **UI 树（脆弱）**：Compose 为了省内存，当页面 A 跳转到页面 B 时，页面 A 的 UI 节点会被**彻底从屏幕上拔掉（Dispose 销毁）**。
*   **内存对象（坚固）**：我们的路由栈 `Navigator.backStack` 本质上就是一个普通的 Kotlin `List<StackEntry>`。只要我们没有执行 `popBackStack()` 把页面踢出列表，这个 `StackEntry` 对象就一直安安稳稳地躺在内存里。

**理解了这 4 点，你就会恍然大悟：为什么我们要让 `StackEntry` 去当那个 Owner！**

## 2. StackEntry：导航栈中的“灵魂容器”

`StackEntry` 是路由框架中最重要的实体。当用户进行一次 `navigateTo` 页面跳转时，框架并不仅仅是把一个 Compose 函数抛给屏幕，而是会为这次跳转创建一个专属的 `StackEntry` 对象。它就像是一个页面的“集装箱”，把页面的配置、运行时的协程、ViewModel 数据全部打包在了一起。

### 逐行代码拆解与设计意图

```kotlin
// StackEntry.kt
// 1. 实现 ViewModelStoreOwner 接口
// 极其重要！实现了这个接口，StackEntry 就拥有了掌管 ViewModel 生杀大权的能力。
data class StackEntry(
    
    // 2. node: 包含路由路径、匹配正则和真正的 Composable 函数的静态节点（来自路由表）
    val node: RouterNode,
    
    // 3. context: 用户跳转时携带的动态数据（例如 /user/123 解析出来的 123 参数，或者额外传递的 extra 对象）
    val context: RouterContext,
    
    // 4. result: 用于处理带有返回值的页面跳转（类似 startActivityForResult）
    val result: DeferredAsyncResult<Any?>? = null,
    
    // 5. id: 为每次页面跳转生成一个全局唯一的 ID，用于 SaveableStateHolder 的状态保存
    val id: String = generateEntryId(),
    
    // 6. preloaderDeferred: 支持页面在做进场动画时，提前并行拉取数据的异步任务
    val preloaderDeferred: Deferred<Any?>? = null
    
) : ViewModelStoreOwner {
    
    // 7. 延迟初始化 _viewModelStore。ViewModelStore 是 Google 官方提供的容器，本质上是一个 HashMap<String, ViewModel>。
    private var _viewModelStore: ViewModelStore? = null

    // 8. 重写 ViewModelStoreOwner 接口的方法，返回专属的 ViewModel 仓库。
    // 设计意图：按需创建。如果这个页面非常简单，没有调用 ViewModel，就不占用内存创建 Store。
    override val viewModelStore: ViewModelStore
        get() {
            if (_viewModelStore == null) {
                _viewModelStore = ViewModelStore()
            }
            return _viewModelStore!!
        }

    // 9. 回调列表：允许开发者在页面彻底销毁时，注册一些自定义的清理逻辑。
    private val disposeCallbacks = mutableListOf<() -> Unit>()

    fun onDispose(callback: () -> Unit) {
        disposeCallbacks.add(callback)
    }

    // 10. dispose()：页面走向生命终点时的“收尸”操作（由 Navigator 在 popBackStack 时调用）。
    internal fun dispose() {
        // 11. 极其关键！调用 viewModelStore 的 clear() 方法。
        // 这会遍历此页面所有的 ViewModel 实例，并触发它们的 onCleared() 方法，让业务层有机会释放网络、数据库连接。
        // 💡 避坑提示：早期版本这里还有一个 `scope?.cancel()`，但因为有了 ViewModelStoreOwner，
        // 开发者直接使用 `viewModelScope` 就能完美管理协程生命周期，所以那个冗余的 scope 已经被移除了。
        _viewModelStore?.clear()
        
        // 12. 依次执行自定义回调，并清空列表。
        disposeCallbacks.forEach { it() }
        disposeCallbacks.clear()
    }

    companion object {
        private var idCounter = 0
        // 15. 全局唯一的 ID 生成器，由计数器和随机数拼接，确保即使用户疯狂进出同一个页面，每次生成的 ID 也绝对不会重复。
        private fun generateEntryId(): String {
            return "entry_${idCounter++}_${kotlin.random.Random.nextInt(10000)}"
        }
    }
}
```

### 如果没有 StackEntry 会发生什么？

#### 1. 灾难一：ViewModel 作用域逃逸（数据污染）
在 Compose 中，当你在页面里调用 `viewModel<MyViewModel>()` 时，系统会顺着 Compose 树向上寻找最近的 `LocalViewModelStoreOwner`。
如果我们的路由框架不提供这个 Owner，系统就会一直往上找，最终找到最外层的 `Activity`（Android）或 `Window`（iOS/Desktop）。

**后果演示（反面教材）**：
假设我们有一个商品详情页，它需要一个 `DetailViewModel` 来加载数据：
```kotlin
@Composable
fun DetailScreen(productId: String) {
    // 如果没有 StackEntry 作为 Owner，这里会一直往上找，找到 Activity 级别的 Owner
    val viewModel: DetailViewModel = viewModel()
    
    LaunchedEffect(productId) {
        viewModel.loadData(productId)
    }
    
    Text("当前商品：${viewModel.productName}")
}
```
**场景**：用户在“商品 A（ID=1）”的详情页，点击了推荐列表里的“商品 B（ID=2）”，于是路由栈变成了：`[DetailScreen(1), DetailScreen(2)]`。
**灾难发生**：因为它们都向同一个 `Activity` 级别的 Owner 请求 `DetailViewModel`，所以页面 B 拿到的 ViewModel 实例，**就是页面 A 刚刚创建的那个实例**！
页面 B 调用 `loadData(2)`，把 ViewModel 里的 `productName` 改成了“商品 B”。
此时用户按返回键回到页面 A，页面 A 重新渲染，读取 `viewModel.productName`，**发现自己变成了“商品 B”！** 数据彻底串台了。

**解法（StackEntry 作为 Owner）**：
当 `StackEntry` 实现了 `ViewModelStoreOwner`，并在 `ScreenRouterRenderer` 中通过 `LocalViewModelStoreOwner provides entry` 注入后：
页面 A 的 `viewModel()` 会向 `StackEntry(A)` 请求实例。
页面 B 的 `viewModel()` 会向 `StackEntry(B)` 请求实例。
它们各自拥有独立的 `ViewModelStore`（本质上是一个 Map），从而完美实现了**页面级的作用域隔离**。

#### 2. 灾难二：抵抗 Compose 的“冷酷无情”（UI 销毁）
我们在前面提到过，Compose 为了省内存，会把不在前台的页面 UI 节点彻底销毁（Dispose）。

**后果演示（反面教材）**：
假设我们不用 `ViewModelStoreOwner`，而是自己用 `remember` 存一个普通的类当 ViewModel：
```kotlin
@Composable
fun UserScreen() {
    // 试图用 remember 保存状态
    val viewModel = remember { UserViewModel() }
    
    LaunchedEffect(Unit) {
        viewModel.fetchUserInfo() // 耗时网络请求
    }
}
```
**场景**：用户在 `UserScreen`，然后跳转到了 `SettingsScreen`。
**灾难发生**：路由栈变成 `[UserScreen, SettingsScreen]`。因为 `UserScreen` 不在栈顶，Compose 为了省内存，会把 `UserScreen` 的 Composable 函数从 UI 树中**彻底移除**。
随着 UI 节点的移除，`remember` 里的 `viewModel` 也会被垃圾回收（GC）。
当用户从 `SettingsScreen` 返回时，`UserScreen` 被重新组合（Recomposition），`remember` 会**重新执行**，创建一个全新的 `UserViewModel`，并**再次发起网络请求**。用户体验极差（每次返回都要重新加载）。

#### 3. 灾难三：状态大混战
由于没有唯一生成的 `id`，如果有两个完全相同的页面 A 叠在一起（比如 `商品详情A` 跳转 `商品详情A` 的变体），Compose 的 `SaveableStateHolder` 会无法区分它们，导致状态错乱。

## 3. TianquViewModel：极其巧妙的“借壳生蛋”

`tianquViewModel` 是开发者在编写 UI 页面时最常用到的 API，但它的源码却短得令人惊讶！只有寥寥几行。

### 逐行代码拆解与设计意图

```kotlin
// TianquViewModel.kt

// 1. inline + reified：内联与具体化类型参数。
// 必须这样写，因为我们要拿到真实的泛型类型 T（例如 UserViewModel），才能向系统申请实例化对应的对象。
@Composable
inline fun <reified T : ViewModel> tianquViewModel(): T {
    
    // 2. 调用官方的 viewModel() 方法，没有多余的一行代码！
    // 看起来极其简单，但背后全赖于架构级的依赖注入！
    return viewModel<T>()
}
```

### 为什么它能生效？（结合前置知识的伟大魔法）

如果你直接在一个光秃秃的 Composable 函数里调用原生的 `viewModel()`，根据**前置知识二**，系统会疯狂向上溯源，最终把这个 ViewModel 绑定到整个 `Activity` 或者整个 `Window` 的根节点上。
这意味着，整个 App 所有的页面，如果你都请求 `UserViewModel`，拿到的竟然是**全 App 唯一的同一个实例**！数据完全串台。

**那么 `tianquViewModel` 为什么不会串台？**
它的魔法并不在这个函数本身，而是在于 **Renderer（渲染器）的配合**。
回想一下我们在 `ScreenRouterRenderer` 中解析的代码，这里用到了**前置知识三（CompositionLocal 魔法）**：
```kotlin
CompositionLocalProvider(
    LocalViewModelStoreOwner provides entry, // 这里的 entry 就是 StackEntry！
) {
    // 渲染你的业务页面
    entry.node.composable(entry.context)
}
```

**设计意图揭秘（偷梁换柱）：**
框架在执行业务的 UI 页面之前，提前在 Compose 的上下文环境里开启了“广播”，把全局的 `ViewModelStoreOwner` 替换成了当前专属于该页面的 **`StackEntry`**！

所以，当开发者在页面中调用 `tianquViewModel()` 时：
1. 表面上，它调用的是官方跨平台的原生 `viewModel()` API。
2. 底层里，系统向上寻找 Owner，立刻就听到了 `CompositionLocalProvider` 的广播，撞上了刚刚注入的 `StackEntry`。
3. 系统顺理成章地向 `StackEntry` 内部的那个大 Map（`viewModelStore`）索要实例。
4. 根据**前置知识四**，即使页面 UI 被 Compose 销毁了，只要 `StackEntry` 还在路由栈的 List 里，这个大 Map 就还在，ViewModel 的数据就完好无损！

### 为什么还要单独包一层 `tianquViewModel`？
既然底层都是调用官方的 `viewModel()`，为什么不直接让开发者写 `viewModel()` 呢？
1. **语义化与心智模型**：路由框架希望告诉开发者，这是一个由 天衢 框架管理生命周期（与导航栈绑定）的 ViewModel，强调其“页面级隔离”的特性。
2. **为未来扩展留后路**：目前直接使用了原生的 `viewModel()`。如果在未来需要集成特定的依赖注入框架（如 Koin、Kodein ），或者我们需要给 ViewModel 传递路由里带过来的特殊参数（RouterContext），我们只需要改这一个函数，而不用修改所有业务页面的代码，极大地增强了框架的**兼容性和可维护性**。
